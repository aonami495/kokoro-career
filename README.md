# ココロキャリア

**障がい者と企業をつなぐ、配慮マッチング型求人プラットフォーム**

> **開発ステータス：MVP（実験段階）**
> 基本機能は実装済みですが、実際の利用シナリオに沿った検証・改善は途上です。
> 現時点で把握している課題と改善方針は「[既知の課題と今後の改善予定](#既知の課題と今後の改善予定)」に記載しています。

---

## 開発の背景

いとこが就労支援の仕事をしており、発達障害やうつ病などを抱える方が社会復帰できる仕組みがあることを知りました。

一方で、既存の求人サービスでは「この職場が自分に合うかどうか」を事前に判断するのが難しいという声があることも知りました。障がいの種類や程度は人それぞれで、必要な配慮も異なります。企業側も「どんな配慮ができるのか」を伝える手段が限られています。

**求職者が必要な配慮と、企業が提供できる配慮を可視化し、ミスマッチを防ぐ仕組みが作れないか。** そう考えて開発したのがココロキャリアです。

---

## 開発手法：AIとの協働開発

このプロジェクトは **Claude Code（Claude Opus 4.5）** を活用して開発しました。

### 自分が担当した部分
- **テーマ選定**：就労支援の現場で感じた課題から、マッチングサービスというアイデアを着想
- **仕様設計**：配慮タグを「必須」と「希望」に分け、重み付けでマッチ度を算出するコンセプトを設計
- **機能要件の定義**：体験実習→日報→本採用という実際の就労移行を反映したフロー設計
- **テスト・動作確認**：各機能の動作検証とバグの発見・修正指示

### AIが担当した部分
- コードの実装（Rails・Hotwire・Tailwind CSS）
- データベース設計の具体化
- テストコードの生成

### この経験から学んだこと
AIなしの自分のスキルレベルは、シンプルなCRUDアプリを作る程度です。しかし「今の自分にできる範囲」に留まるのではなく、AIを道具として使いこなすことで、より複雑で社会的に意義のあるプロダクトに挑戦できることを実感しました。同時に、AIが出力したコードをそのまま使うのではなく、なぜそう書くのかを理解し、自分でも書けるようになるための学習を続けています。

## 主な特徴

### 配慮タグによるマッチング
求職者が「必須の配慮」と「あると嬉しい配慮」を登録し、企業が「提供可能な配慮」を登録。双方の情報をもとにマッチ度をスコア化します。

### マッチングスコアの設計思想

```
マッチ度 = (必須配慮のマッチ率 × 70%) + (希望配慮のマッチ率 × 30%)
```

この計算式には意図があります。

- **必須配慮が1つでも欠けていればスコアは0%**：「静かな環境が必須」な方にオープンオフィスの企業を紹介するのは、マッチングとして失格。ここは妥協できないという設計判断です。
- **必須70%・希望30%の重み付け**：必須が全て満たされた上で、希望の一致度で差がつく仕組み。「最低限の安心」を保証した上で「より良い環境」を探せるようにしています。

### 体験実習フロー
実際の就労移行支援では、いきなり採用ではなく「まず職場を体験する」ステップがあります。この現実のフローを反映し、以下の流れを実装しました。

1. 求職者が応募
2. 企業が実習オファーを送信（日程指定）
3. 求職者がオファーを承諾
4. 実習期間中、双方が日報を投稿
5. 実習終了後、企業が「本採用」または「終了」を選択

### 日報システム
実習中の双方向コミュニケーションをサポート。求職者と企業の両方が日報を投稿でき、実習期間中の様子を可視化します。

## セットアップ

### 必要条件
- Ruby 3.2.2
- PostgreSQL 14+
- Node.js 18+

### インストール

```bash
# リポジトリをクローン
git clone https://github.com/aonami495/kokoro-career.git
cd kokoro-career

# 依存関係をインストール
bundle install

# データベースを作成・マイグレーション
bin/rails db:create
bin/rails db:migrate

# シードデータを投入（配慮タグ）
bin/rails db:seed

# Tailwind CSSをビルド
bin/rails tailwindcss:build

# サーバーを起動
bin/rails server
```

### 開発用アカウント

シードデータ投入後、以下のコマンドで管理者アカウントを作成できます：

```bash
bin/rails runner "User.create!(name: '管理者', email: 'admin@example.com', password: 'password', user_type: :admin)"
```

## 既知の課題と今後の改善予定

### 現時点で把握している課題

このアプリはAIとの協働開発でどこまで作れるかを実験する目的で始めたプロジェクトです。MVP として基本機能は動作しますが、実際に使ってみると以下の課題があります。

- **実利用を想定したバリデーションが不十分**：入力値のチェックやエラーハンドリングが甘い箇所がある
- **UIの導線が直感的でない部分がある**：初めて使うユーザーが迷わず操作できるかの検証ができていない
- **テストカバレッジが不十分**：主要な機能パスはテストしているが、エッジケースの考慮が足りない
- **パフォーマンスの検証ができていない**：配慮タグが増えた場合のマッチング計算速度を検証していない

### 改善予定

- [ ] 実際のユースケースに沿ったE2Eテストの追加
- [ ] バリデーション・エラーハンドリングの強化
- [ ] スクリーンショットの追加（求職者画面・企業画面・マッチング結果画面）
- [ ] マッチングアルゴリズムの改善（配慮タグ間の関連性を考慮した重み付け）
- [ ] 求職者プロフィールの充実（職歴・スキル・希望条件の追加）
- [ ] 企業レビュー機能（実習経験者からのフィードバック）
- [ ] レスポンシブデザインの改善
- [ ] CI/CDパイプラインの構築
